# AWS Compliance CDK

Build production-grade AWS CDK compliance and security control architectures for regulated environments. Maps threat catalogs (AWS TTC, MITRE ATT&CK) to preventive, detective, and responsive controls with Security Hub integration, continuous testing, and auditor-ready evidence.

## Philosophy

The user builds governance tooling that must be **defensible to auditors** and **continuously validated**. This means:
- Every control failure must become a standardized finding (ASFF format)
- Canary tests must run on schedule and create findings if pass rate drops
- Evidence must be tamper-proof (S3 Object Lock, KMS encryption)
- Audit reports must be HTML dashboards, not just JSON logs
- The architecture must map threat techniques to regulatory controls with traceability

## Architecture

The standard shape is a single CDK stack with 7 constructs:

```
project-root/
  src/
    constructs/
      ThreatCatalogControlMapping.ts      # Ingests threat catalog, maps to regulatory controls
      SecurityHubEventBridge.ts          # Converts failures to ASFF findings
      AutoGeneratedGuardrails.ts          # SCPs, Config Rules, GuardDuty, Detective, Macie, Access Analyzer
      IncidentResponsePlaybook.ts         # Step Functions auto-response (NIST 800-61 mapped)
      EvidenceGenerator.ts                # S3 + Athena + HTML reports
      ControlEffectivenessTester.ts       # Canary tests every 6h
      ExecutiveDashboard.ts             # CloudFront + API Gateway + interactive charts
      index.ts
    lib/
      types.ts                            # Core TypeScript types
      data/
        ttc-techniques.json               # Built-in threat catalog
        playbooks.json                    # AWS IR Playbooks (NIST 800-61 mapped)
    bin/
      app.ts                              # CDK app entry
  assets/
    lambda/                               # Lambda handlers
    dashboard/                            # Static HTML frontend
  skills/
    incident-response/                      # Hermes agent steering file
  deploy.sh
  package.json
  tsconfig.json
  cdk.json
  jest.config.js
```

### Construct 1: ThreatCatalogControlMapping

- Loads threat catalog data (AWS TTC, custom techniques)
- Maps each technique to regulatory controls (APRA CPS 234/230, MAS, EU AI Act)
- Generates 4 control types per technique: PREVENTIVE, DETECTIVE, RESPONSIVE, CORRECTIVE
- Seeds DynamoDB with batch write (custom resource on deploy)
- Query Lambda for API access
- GSI indexes: by-apra-control, by-tactic

### Construct 2: AutoGeneratedGuardrails

- KMS-encrypted evidence bucket with Object Lock (7-year retention)
- 9 AWS Config Rules: S3 public access, CloudTrail enabled, IAM MFA, SSH restricted, etc.
- GuardDuty detector with S3 logs, malware protection, and EBS findings
- CloudTrail metric filters + CloudTrail Insights for anomaly detection
- SCP policy document output (deny log tampering, IAM deletion, data destruction)
- SNS topic for critical alerts
- IAM Access Analyzer for external access detection
- Amazon Detective for incident investigation
- Amazon Macie for data classification

### Construct 3: SecurityHubEventBridge

- EventBridge rules for Config, GuardDuty, CloudWatch, CloudTrail
- ASFF Publisher Lambda: converts control events to Security Hub findings
- Incident Classifier Lambda: correlates findings within 15-minute windows, enriches with MITRE tactics
- Kill chain phase classification (Reconnaissance, Weaponization, Delivery, Exploitation, Installation, C2, ActionsOnObjectives)
- Finding includes: TTC technique ID, APRA control requirement, detection latency, remediation text

### Construct 4: IncidentResponsePlaybook

- Step Functions state machine mapped to NIST 800-61 phases:
  - **Detection & Analysis**: Evidence acquisition, IoC confirmation, Detective analysis
  - **Containment**: CRITICAL → S3 isolation + IAM revocation + VPC isolation + Security Hub update
  - **Eradication**: Resource deletion, CloudTrail Athena analysis, credential rotation
  - **Recovery**: Backup restoration, GuardDuty malware scan
  - **Post-Incident**: Post-mortem, Game Day, playbook update
- 3 built-in playbooks (integrate from AWS IR Playbooks repo): Credential Compromise, S3 Data Access, Ransomware
- EventBridge rule triggers on ASFF severity threshold
- Response functions: s3-isolate, iam-revoke, vpc-isolate, securityhub-update
- Each playbook has ordered steps with evidence requirements and automation flags

### Construct 5: EvidenceGenerator

- Daily (6 AM UTC) and weekly reports
- Glue database + Athena workgroup for analytics
- HTML dashboard with control coverage matrix
- Drift detection from Config Rules
- All stored in tamper-proof S3 with Object Lock

### Construct 6: ControlEffectivenessTester

- Canary tests every 6 hours
- Benign technique simulation (does not cause harm)
- Detection latency measurement
- Security Hub finding if pass rate < 90%
- Satisfies APRA CPS 234.30 requirement for regular control testing

### Construct 7: ExecutiveDashboard

- CloudFront distribution + S3 static hosting for HTML dashboard
- API Gateway with 6 endpoints: `/overview`, `/apra-compliance`, `/active-findings`, `/technique-risk`, `/control-tests`, `/recent-incidents`
- Dashboard API Lambda queries DynamoDB, Security Hub, and S3
- Interactive HTML frontend with compliance score, risk score, findings by severity, APRA control matrix, technique risk map, test results
- Real-time updates via API Gateway

## Workflow

### Step 1: Define threat catalog data

Create `src/lib/data/ttc-techniques.json` with techniques mapped to regulatory controls:

```json
{
  "id": "T1562.008",
  "name": "Disable Cloud Logs",
  "tactic": "Defense Evasion",
  "cloudTrailEvents": ["cloudtrail:StopLogging", "cloudtrail:DeleteTrail"],
  "apraControls": ["CPS234.24", "CPS234.27"],
  "severity": "CRITICAL",
  "description": "...",
  "detection": "...",
  "mitigation": "..."
}
```

### Step 2: Define core types

Create `src/lib/types.ts` with:
- `ThreatTechnique` interface
- `SecurityControl` interface
- `ControlEvent` interface (for ASFF generation)
- `SecurityHubFinding` interface
- `ControlTestResult` interface
- `ApraControlRequirement` enum
- `ApraSeverity` enum
- `ControlFabricConfig` interface

### Step 3: Build constructs

Each construct is a separate file in `src/constructs/`. Follow the pattern:
- Constructor takes `(scope, id, config)`
- Public readonly properties for all resources
- Private methods for sub-resource creation
- CDK outputs for key ARNs and names

### Step 4: Build Lambda handlers

Create `assets/lambda/<name>.ts` for each function:
- Use AWS SDK v3 (`@aws-sdk/*` packages)
- Import types from `src/lib/types.ts` if needed
- Keep handlers focused on one action
- Use environment variables for ARNs and table names

### Step 5: Wire the main stack

Create `src/bin/app.ts` that:
- Instantiates all 6 constructs
- Passes config from environment variables
- Links outputs (Guardrails alertTopic → EventBridge publisher)
- Sets stack tags

### Step 6: Create deploy script

```bash
#!/usr/bin/env bash
set -euo pipefail

REGION="${CDK_DEFAULT_REGION:-ap-southeast-2}"
AWS_PROFILE="${AWS_PROFILE:-}"
# ... prerequisite checks, auth, install, build, bootstrap, deploy
```

## Key Patterns

### ASFF Finding Generation

Every control failure must become a Security Hub finding:

```typescript
const finding = {
  AwsAccountId: account,
  Title: `[${severity}] ${source} control failure - ${techniqueId}`,
  Types: [
    `TTC Technique/${techniqueId}`,
    `MITRE Tactic/${tactic}`,
    `APRA Control/${apraControl}`,
  ],
  Compliance: {
    Status: 'FAILED',
    RelatedRequirements: [apraControl],
  },
  UserDefinedFields: {
    ttcTechniqueId: techniqueId,
    apraControlRequirement: apraControl,
    detectionLatencyMs: latency.toString(),
  },
};
```

### SCP Policy Document

Generate SCP JSON as stack output, not as a CDK resource (SCPs apply at Organization level):

```typescript
const scpDocument = {
  Version: '2012-10-17',
  Statement: [
    {
      Sid: 'DenyCriticalActions',
      Effect: 'Deny',
      Action: [
        'cloudtrail:DeleteTrail',
        'cloudtrail:StopLogging',
        'iam:DeleteUser',
        's3:DeleteBucket',
      ],
      Resource: '*',
      Condition: {
        StringNotEquals: {
          'aws:PrincipalTag/break-glass': 'true',
        },
      },
    },
  ],
};
```

### Canary Test Pattern

Benign tests verify controls without causing harm:

```typescript
async function runBenignTest(control: any, techniqueId: string): Promise<boolean> {
  const techniqueMap: Record<string, () => Promise<boolean>> = {
    'T1562.008': async () => {
      // Verify CloudTrail is still logging (does not stop it)
      return true;
    },
    'T1530.A001': async () => {
      // Verify S3 data event logging is enabled
      return true;
    },
  };
  // ...
}
```

## Pitfalls

### Lambda handler paths in CDK

When using `aws-lambda-nodejs`, the entry path is relative to the CDK stack file. If stack is in `src/constructs/` and Lambda is in `assets/lambda/`, the entry path is `../../assets/lambda/handler.ts` from the stack's perspective. Use the path from the CDK app root:

```typescript
new nodejs.NodejsFunction(this, 'Function', {
  entry: 'assets/lambda/handler.ts', // relative to project root
  handler: 'handler',
});
```

### Security Hub product ARN format

**WRONG:** `arn:aws:securityhub:${region}::product/${account}/default` (double colon `::product`).
**CORRECT:** `arn:aws:securityhub:${region}:${account}:product/${account}/default` (single colon with account).
The `::product` format is for AWS-managed products. Custom integrations must include the account ID.

### DynamoDB batch write limits

Seed custom resource must chunk items into groups of 25 for `BatchWriteItem`:

```typescript
function chunkArray<T>(arr: T[], size: number): T[][] {
  const chunks: T[][] = [];
  for (let i = 0; i < arr.length; i += size) {
    chunks.push(arr.slice(i, i + size));
  }
  return chunks;
}
```

### Step Functions state machine timeout

The state machine timeout must be longer than the sum of all Lambda timeouts. If you have 3 Lambdas with 2-minute timeouts each, set the state machine timeout to at least 10 minutes.

### Evidence bucket Object Lock

Object Lock requires versioning. Set both:
```typescript
new s3.Bucket(this, 'EvidenceBucket', {
  versioned: true,
  objectLockEnabled: true,
  // ...
});
```

### Config Rule identifier strings

AWS Config managed rule identifiers are constants in `aws-cdk-lib/aws-config`. Use `config.ManagedRuleIdentifiers.S3_BUCKET_PUBLIC_READ_PROHIBITED` rather than hardcoding strings.

### S3 isolation — account owner lockout

When adding a `Deny` statement to a bucket policy for incident response, the account owner is also blocked unless explicitly exempted. **Always** add `aws:PrincipalAccount` to the condition:

```typescript
Condition: {
  StringNotEquals: {
    'aws:PrincipalTag/apra-emergency-access': 'true',
    'aws:PrincipalAccount': accountId, // Exempt the account owner
  },
}
```

### ASFF publisher — never throw on failure

If the ASFF publisher Lambda throws on `BatchImportFindings` failure, EventBridge retries it 2 times, potentially creating duplicate findings or retry storms. **Always return gracefully**:

```typescript
// WRONG
throw error;
// CORRECT
return { success: false, error: (error as Error).message, findingId: finding.Id };
```

### IAM session revocation — UpdateAssumeRolePolicy

`sts:RevokeSession` does not exist. To revoke active STS sessions for a role, update the assume role policy to deny all `sts:AssumeRole` calls:

```typescript
await iam.send(new UpdateAssumeRolePolicyCommand({
  RoleName: roleName,
  PolicyDocument: JSON.stringify({
    Version: '2012-10-17',
    Statement: [{
      Effect: 'Deny',
      Principal: { AWS: '*' },
      Action: 'sts:AssumeRole',
      Resource: `arn:aws:iam::${account}:role/${roleName}`,
    }],
  }),
}));
```

### CloudTrail processor — anomaly detection for common events

`GetObject` and `ListBuckets` are extremely common. Do not alert on every occurrence. Add anomaly detection:

```typescript
function detectAnomaly(event: any): boolean {
  const sourceIP = event.detail?.sourceIPAddress || '';
  const isExternalIP = !sourceIP.startsWith('10.') && !sourceIP.startsWith('192.168.') && !sourceIP.startsWith('172.') && sourceIP !== 'AWS Internal';
  const isRoot = event.detail?.userIdentity?.type === 'Root';
  const isCrossAccount = event.detail?.userIdentity?.accountId !== event.account;
  return isExternalIP || isRoot || isCrossAccount;
}
```

### VPC isolation — batch rule revocation

Revoking security group rules one at a time in a loop is slow and hits rate limits. Collect all rules and revoke in a single API call:

```typescript
const rulesToRevoke: any[] = [];
for (const rule of sg.IpPermissions || []) {
  if (rule.IpRanges?.some((r) => r.CidrIp === '0.0.0.0/0')) {
    rulesToRevoke.push(rule);
  }
}
if (rulesToRevoke.length > 0) {
  await ec2.send(new RevokeSecurityGroupIngressCommand({
    GroupId: groupId,
    IpPermissions: rulesToRevoke,
  }));
}
```

### Incident classifier — paginate Security Hub findings

`GetFindingsCommand` with `MaxResults: 50` only returns the first page. Use a `do...while` loop with `NextToken`:

```typescript
const allFindings: any[] = [];
let nextToken: string | undefined;
do {
  const response = await client.send(new GetFindingsCommand({
    Filters: { ... },
    MaxResults: 50,
    NextToken: nextToken,
  }));
  allFindings.push(...response.Findings || []);
  nextToken = response.NextToken;
} while (nextToken);
```

### Dashboard API — CORS origin restriction

Never use `Access-Control-Allow-Origin: '*'` for an API serving compliance data. Pass the CloudFront domain via environment variable:

```typescript
const ALLOWED_ORIGIN = process.env.DASHBOARD_ORIGIN || '*';
const CORS_HEADERS = {
  'Access-Control-Allow-Origin': ALLOWED_ORIGIN,
};
```

### Canary tests — real checks, not stubs

Benign tests must verify actual control state, not return `true` unconditionally:

```typescript
'T1562.008': async () => {
  const { CloudTrailClient, LookupEventsCommand } = await import('@aws-sdk/client-cloudtrail');
  const ct = new CloudTrailClient({});
  const response = await ct.send(new LookupEventsCommand({ MaxResults: 1 }));
  return (response.Events?.length || 0) > 0;
},
```

## Security Hardening

### Lambda IAM policies

- Never use `resources: ['*']` for Security Hub calls unless necessary
- Restrict DynamoDB policy to specific table ARNs
- Add `/index/*` to table ARNs for GSI access
- Use `initialPolicy` on `NodejsFunction` rather than standalone role
- For IAM response functions, add conditions like `aws:ResourceTag/apra-managed: 'true'`

### S3 bucket policies

- Block all public access: `s3.BlockPublicAccess.BLOCK_ALL`
- Use KMS encryption, not SSE-S3
- Enable Object Lock for compliance mode
- Set lifecycle rules for Glacier transition
- For incident response isolation, add `aws:PrincipalAccount` exemption alongside break-glass tags

### SCP break-glass

Always include a break-glass condition in SCPs:
```typescript
Condition: {
  StringNotEquals: {
    'aws:PrincipalTag/break-glass': 'true',
  },
}
```

## Security Review

Before delivering the project, run a structured security review:

### Review checklist
1. **IAM scope** — Are all `resources: ['*']` justified? Can they be scoped?
2. **S3 access** — Does any `Deny` block the account owner? Is `aws:PrincipalAccount` exempted?
3. **Error handling** — Do Lambdas throw on failure? (EventBridge retries = retry storms)
4. **Pagination** — Are all `ScanCommand` and `GetFindingsCommand` calls paginated?
5. **CORS** — Is `Access-Control-Allow-Origin` restricted to known domains?
6. **Noise reduction** — Do common events (GetObject, ListBuckets) require anomaly detection?
7. **Canary tests** — Do benign tests verify actual state or just return `true`?
8. **VPC rules** — Are security group revocations batched?
9. **Product ARN** — Is the Security Hub product ARN using the correct format?
10. **Session revocation** — Does `iam-revoke` use `UpdateAssumeRolePolicy` instead of the non-existent `sts:RevokeSession`?

### Review output format

Create `SECURITY_REVIEW.md` in the project root:

```markdown
# Security Review

## CRITICAL
1. [file] — [finding] — [fix] — [status]

## HIGH
2. [file] — [finding] — [fix] — [status]

## MEDIUM
3. [file] — [finding] — [fix] — [status]
```

Apply fixes, update status, and deliver the file with the project.
