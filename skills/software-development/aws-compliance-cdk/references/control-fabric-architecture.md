# Control Fabric Architecture Details

Component interaction diagram and deployment sequence for the Threat-Aligned Compliance Control Fabric.

## Component Interaction

```
ThreatCatalogControlMapping
  |├── DynamoDB (techniques, controls, APRA mappings)
  |└── Query Lambda (API access)
         |
         v
AutoGeneratedGuardrails
  |├── Config Rules (9 rules)
  |├── GuardDuty Detector (if enabled)
  |├── CloudTrail Metric Filters
  |├── S3 Evidence Bucket (KMS + Object Lock)
  |└── SNS Alert Topic
         |
         v
SecurityHubEventBridge
  |├── EventBridge Rules (control failures, high severity, correlated)
  |├── ASFF Publisher Lambda
  |└── Incident Classifier Lambda
         |
         v
IncidentResponsePlaybook
  |├── Step Functions State Machine
  |├── S3 Isolate Lambda
  |├── IAM Revoke Lambda
  |├── VPC Isolate Lambda
  |└── Security Hub Update Lambda
         |
         v
EvidenceGenerator
  |├── Glue Database
  |├── Athena Workgroup
  |├── Report Generator Lambda
  |└── Drift Detector Lambda
         |
         v
ControlEffectivenessTester
  |├── DynamoDB Test Results
  |├── Canary Runner Lambda
  |└── Test Validator Lambda
```

## Event Flow

1. **Control Failure Detected**
   - Config Rule goes NON_COMPLIANT → EventBridge
   - GuardDuty finding → EventBridge
   - CloudWatch Alarm triggers → EventBridge

2. **ASFF Finding Created**
   - EventBridge routes to `AsffPublisher` Lambda
   - Lambda calls `securityhub:BatchImportFindings`
   - Finding includes TTC technique ID, APRA control, detection latency

3. **Incident Classification**
   - `IncidentClassifier` Lambda runs on HIGH/CRITICAL findings
   - Correlates with other findings in 15-minute window
   - Determines kill chain phase
   - Enriches finding with correlation data

4. **Auto-Response (if enabled)**
   - EventBridge triggers Step Functions state machine
   - CRITICAL: all 4 response actions
   - HIGH: approval window + IAM revocation
   - Updates Security Hub finding status

5. **Evidence Generation**
   - Daily at 6 AM UTC: report generator runs
   - Produces HTML dashboard with control coverage matrix
   - Stores in S3 with Object Lock

6. **Canary Testing**
   - Every 6 hours: canary runner executes
   - Simulates benign technique versions
   - Measures detection latency
   - If pass rate < 90%, creates Security Hub finding

## Deployment Sequence

```bash
# 1. Bootstrap
npx cdk bootstrap

# 2. Deploy (creates all resources)
npx cdk deploy

# 3. Apply SCP (manual step)
# Copy SCP policy document from stack outputs
# Apply to Organization root or target OU

# 4. Enable Security Hub (if not already enabled)
aws securityhub enable-import-findings

# 5. Verify (wait 6 hours for first canary test)
aws dynamodb scan --table-name <TestResultsTable>
aws securityhub get-findings --filters "ProductArn=<productArn>"
```

## Key Configuration

| Parameter | Default | Impact |
|---|---|---|
| `ENABLE_GUARDDUTY` | `true` | Creates GuardDuty detector |
| `ENABLE_AUTO_RESPONSE` | `false` | Enables Step Functions auto-response |
| `AUTO_RESPONSE_THRESHOLD` | `HIGH` | Minimum severity for auto-response |
| `EVIDENCE_RETENTION_DAYS` | `2555` | 7 years for S3 lifecycle |

## Outputs

| Output | Description |
|---|---|
| `MappingTableName` | DynamoDB table for TTC-to-APRA mappings |
| `EvidenceBucketName` | S3 bucket for audit evidence |
| `AlertTopicArn` | SNS topic for critical alerts |
| `AsffPublisherArn` | Lambda that publishes to Security Hub |
| `ResponseStateMachineArn` | Step Functions for incident response |
| `TestResultsTableName` | DynamoDB table for canary test results |
